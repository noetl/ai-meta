snippets, hours added to locations/hotels/activities widgets.

Auth model — PATTERN C HYBRID
-----------------------------
Two distinct credential paths, each with the minimum blast radius
for what it has to do:

  1. **Backend Places + Routes API calls** (search_text, place_details,
     nearby_search; plus any Routes API calls if added later) →
     authenticated via **Workload Identity** on the worker SA
     (`noetl-worker-mcp@noetl-demo-19700101.iam.gserviceaccount.com`).
     No secret to manage. Same `google.auth.default(scopes=[cloud-platform])`
     pattern that mcp/vertex-ai.yaml already uses. Sent as
     `Authorization: Bearer <token>` + `X-Goog-User-Project:
     noetl-demo-19700101` + `X-Goog-FieldMask: <fields>`.

  2. **Widget-embedded image URLs** (Maps Static API + Place Photos) →
     authenticated via a **restricted API key** stored in Secret Manager
     as `google-maps-widget-key`. The key has HTTP-referrer restriction
     (mestumre.dev/* + gateway.mestumre.dev/*) + API restriction (Maps
     Static + Places API New only) + per-day quota cap. The worker
     reads the key from Secret Manager and embeds it in URLs the browser
     fetches: `...&key=<widget-key>`. SA OAuth doesn't work here because
     the browser can't carry a SA token; widget keys are unavoidable for
     image URLs but their blast radius is minimal (referrer-gated, image
     endpoints only, quota-capped).

Architectural rules:
  - SUPPLEMENTARY layer, NOT a replacement for Amadeus.
  - Default OFF (opt-in via workload field).
  - Per-execution cap: max 10 enrichments.
  - Enrichment failure is non-blocking — fall back to Amadeus-only widget.
  - No removal of existing Amadeus paths.
  - Free-tier disciplined: GCP-level quota cap is the hard ceiling.
  - **Pattern C is mandatory**: backend tools MUST use SA OAuth; widget
    URL builders MUST use the widget key. Don't merge the two.
  - Don't provision the widget API key or modify SA IAM bindings yourself.
  - No release cut. No git push from ai-meta.
