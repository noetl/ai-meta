Context:
  Amadeus test API has been returning HTTP 500 on flights/locations endpoints
  persistently since the multi-provider round (May 9). Friendly-error widget
  masks it — graceful degradation works — but we have zero diagnosis. Audit
  side-effect round just gave us concrete data: every render_amadeus_failure
  row in travel_agent_events has the originating intent, the
  envelope_status_code, and the failure envelope. Time to use it.

Investigation strategy:

  1. Characterise the failure pattern via SQL on travel_agent_events.
     Group render_amadeus_failure rows by intent + status_code over the last
     7 days. Pull 3-5 full payloads. Note pattern (intent-specific /
     endpoint-wide / time-clustered / status-code distribution).

  2. Reproduce a representative 500 via direct curl using the EXACT payload
     shape mcp/amadeus sends. Capture full response body — Amadeus typically
     includes errors[] with status/code/title/detail. SANITISE tokens
     throughout (first 6-10 chars max).

     Sanity check: also run curl against a known-good Amadeus example payload
     from their docs. If that ALSO returns 500, the sandbox is flaky.

  3. Compare our payload against the current Amadeus self-service spec.
     Diff required vs optional, formats, path/version, headers.

  4. Verdict:
     (a) Our payload has a fixable bug → small ops PR, re-smoke.
     (b) Sandbox flake → no PR, document, recommend production endpoint
         switch as future round.
     (c) Auth/credential quirk → no PR this round, document, propose
         token-refresh follow-up.
     Pick the smallest correct verdict.

  5. Write sync issue (sync/issues/2026-05-11-amadeus-test-api-500s.md)
     with characterisation, curl reproduction (sanitised), verdict,
     recommendations. Plus a memory entry.
