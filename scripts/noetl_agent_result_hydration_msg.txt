Context:
  Item #7's Amadeus investigation surfaced a real engine bug: when agent →
  MCP sub-executions return large results, the agent executor's
  `envelope.data` ends up being the TRUNCATED `/api/executions/{id}/status`
  doc instead of the actual MCP envelope. `_compact_status_variables` in
  noetl/server/api/core/utils.py replaces values over _STATUS_VALUE_MAX_BYTES
  with `{_truncated, _original_size_bytes, _preview}` stubs. Travel runtime's
  `when: amadeus_via_mcp_activities.data.ok | default(false)` evaluates
  against the stub (no .ok field) → routes successful executions to
  render_amadeus_failure widget.

  The travel runtime ALREADY works around this for Vertex AI and Ollama via
  `_fetch_vertex_child_context` / `_fetch_ollama_child_context` helpers that
  walk events directly. This round moves the hydration into the noetl agent
  executor so all callers (Amadeus activities, vertex, ollama, future MCP
  hops) benefit.
