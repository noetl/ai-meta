Round goal
----------
Add Duffel (https://duffel.com) as the new default flights provider on the
travel agent. Introduce a `flight_provider: duffel | amadeus` selector on
the travel runtime so operators can flip back to Amadeus per-workload. New
default is `duffel`. Search-only first cut; booking/payment deferred to
Thread 3 (payment + monetization).

Why Duffel over Amadeus as default
----------------------------------
- Duffel search is $0 in both test and live environments.
- Amadeus test API has a persistent 5xx flake (sync/issues/2026-05-11-amadeus-test-api-500s.md)
  that surfaced again as the locations/hotels AMBER on the Pattern C round.
- Duffel auth is a single static bearer token per env — much simpler than
  Amadeus's OAuth client-credentials dance with per-env credentials.
- 200+ airline coverage via NDC + direct connections + selective GDS.

Locations / hotels / activities remain on Amadeus regardless of
`flight_provider`. Only the flights intent is touched.

Scope locked
------------
- Search-only. NO create_order, NO seat maps, NO payment tools.
- Test environment only. `duffel_live_token_path` is a placeholder; do not
  exercise it.
- Default `flight_provider: duffel`. Amadeus stays as opt-out via
  `flight_provider: amadeus`.
- `search_places` is flight origin/destination autocomplete only. The
  locations widget keeps using Amadeus `search_locations`.

Tool surface (mcp/duffel.yaml)
------------------------------
Four tools, MCP `tools/list` + `tools/call` dispatch shape mirroring
`automation/agents/mcp/amadeus.yaml`:

  - **search_offers**: POST /air/offer_requests?return_offers=true.
    Synchronous flight search. Normalize result into the Amadeus-style
    envelope so `render_flights` doesn't need shape-aware branching.

  - **get_offer**: GET /air/offers/{id}. Canonical offer detail.

  - **search_places**: GET /places/suggestions?query=. Airport/city
    autocomplete for flight inputs.

  - **get_airlines**: GET /air/airlines?limit=200. Reference data.

Duffel auth — single bearer token
---------------------------------
- Kadyapam provisions `duffel-api-test` in GCP Secret Manager out-of-band
  (recipe in bridge task's `pre_handoff_for_kadyapam.step_2`).
- The mcp/duffel playbook reads the token via the keychain pattern that
  mcp/amadeus already uses for its OAuth credentials — but here the secret
  IS the bearer token (no token-exchange step).
- Every HTTP call carries:
    Authorization: Bearer <token>
    Accept: application/json
    Duffel-Version: v2
    Content-Type: application/json   (on POST)
- Test/live env is determined by which token sits behind the secret. Both
  envs hit the SAME `https://api.duffel.com` base URL — only the token
  prefix differs (`duffel_test_*` vs `duffel_live_*`). The `duffel_env`
  workload field selects which secret path to read; it does NOT switch
  the base URL.

Travel runtime integration
--------------------------
`repos/ops/automation/agents/travel/runtime.yaml`:

  1. New workload field `flight_provider: duffel` (valid values:
     `duffel | amadeus`; default `duffel`).
  2. Workload field `duffel_env: test` (default test; live is a future
     placeholder).
  3. Flights step branches on `workload.flight_provider`:
     - `duffel` → hop into `mcp/duffel` with `tool: search_offers`.
     - `amadeus` → hop into `mcp/amadeus` with `tool: search_flights`
       (existing behavior).
  4. `mcp:duffel` appended to capabilities.

The Duffel MCP playbook normalizes Duffel offers into the Amadeus-style
shape inside `search_offers`, so `render_flights` stays unchanged. Spec
for the normalized envelope is in the bridge task under
`design_brief_for_codex.travel_runtime_integration.5`.
