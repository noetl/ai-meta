## Context

After the GKE Option-A decision, the Helm chart at
`repos/ops/automation/helm/noetl/` is the canonical GKE deploy
spec. The chart already has a `worker-keda-scaledobject.yaml`
template, but it emits a **Prometheus** trigger that doesn't
match how we actually scale workers (queue-driven via NATS
JetStream). The real KEDA scaler in GKE is the **external** file
at `repos/ops/ci/manifests/keda/scaledobject-worker-cpu-01.yaml`,
applied manually with live patches because the file's committed
defaults are kind-shaped (`account: NOETL`,
`nats.nats.svc.cluster.local:8222`) and GKE's Helm NATS is
different-shaped (`account: $G`,
`nats-headless.nats.svc.cluster.local:8222`).

Goal of this round: make the chart emit a NATS-JetStream
ScaledObject for GKE with the right Helm-NATS values, migrate
the live cluster onto it, leave the kind-profile external file
alone.

## What this round delivers

1. **Chart change:** rewrite
   `repos/ops/automation/helm/noetl/templates/worker-keda-scaledobject.yaml`
   to emit a `nats-jetstream` trigger; add corresponding defaults
   under `worker.autoscaling.keda.nats.*` in `values.yaml`.
2. **GKE provision playbook:** flip the default so GKE
   provisioning opts into chart-templated KEDA
   (`worker.autoscaling.keda.enabled=true` for the GKE profile).
3. **Cluster migration (pre-authorized):** delete the external
   `noetl-worker-scaler-worker-cpu-01` ScaledObject + its HPA;
   `helm upgrade --reuse-values --set worker.autoscaling.keda.enabled=true`;
   verify the chart-templated ScaledObject + its HPA are the only
   ones present.
4. **Open ops PR** with the chart + playbook changes; do not
   merge.
5. **Result file** at
   `handoffs/active/2026-05-24-keda-chart-template/round-01-result.md`.

### Target chart template shape

The new template must produce (with values substituted):

```yaml
{{- if and .Values.worker.autoscaling.enabled .Values.worker.autoscaling.keda.enabled }}
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: noetl-worker
  namespace: {{ .Values.namespace }}
  labels:
    app: noetl-worker
    component: worker
    managed-by: helm
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: noetl-worker
  pollingInterval: {{ .Values.worker.autoscaling.keda.pollingInterval }}
  cooldownPeriod: {{ .Values.worker.autoscaling.keda.cooldownPeriod }}
  minReplicaCount: {{ .Values.worker.autoscaling.minReplicas }}
  maxReplicaCount: {{ .Values.worker.autoscaling.maxReplicas }}
  triggers:
    - type: nats-jetstream
      metadata:
        natsServerMonitoringEndpoint: {{ .Values.worker.autoscaling.keda.nats.monitoringEndpoint | quote }}
        account: {{ .Values.worker.autoscaling.keda.nats.account | quote }}
        stream: {{ .Values.worker.autoscaling.keda.nats.stream | quote }}
        consumer: {{ .Values.worker.autoscaling.keda.nats.consumer | quote }}
        lagThreshold: {{ .Values.worker.autoscaling.keda.nats.lagThreshold | quote }}
        activationLagThreshold: {{ .Values.worker.autoscaling.keda.nats.activationLagThreshold | quote }}
        useHttps: {{ .Values.worker.autoscaling.keda.nats.useHttps | quote }}
{{- end }}
```

Note the **outer guard requires BOTH** `worker.autoscaling.enabled`
**and** `worker.autoscaling.keda.enabled` to be true. This is
deliberate: the existing `worker-hpa.yaml` guard is
`enabled AND NOT keda.enabled`, so the two templates are mutually
exclusive — you cannot accidentally render both.

> ⚠️ Confirm the existing `worker-hpa.yaml` guard still reads
> `{{- if and .Values.worker.autoscaling.enabled (not .Values.worker.autoscaling.keda.enabled) }}`.
> If anything has drifted, stop and flag in the result instead of
> rewriting both templates in this round.

### Target values.yaml shape

Under the existing `worker.autoscaling` block, expect something like:

```yaml
worker:
  autoscaling:
    enabled: false              # default; kind playbook leaves this off
    minReplicas: 1
    maxReplicas: 20             # bumped from 8 for KEDA profile
    targetCPUUtilizationPercentage: 70
    keda:
      enabled: false            # opt-in (GKE provision flips on)
      pollingInterval: 10
      cooldownPeriod: 30
      nats:
        account: "$G"
        monitoringEndpoint: nats-headless.nats.svc.cluster.local:8222
        stream: NOETL_COMMANDS
        consumer: noetl_worker_pool
        lagThreshold: "10"
        activationLagThreshold: "1"
        useHttps: "false"
```

If the existing values block already has `keda:` keys (e.g.
`pollingInterval`, `cooldownPeriod`, the Prometheus keys
`serverAddress`/`metricName`/`query`/`threshold`/`activationTargetQueryValue`):
- **Drop the Prometheus keys** (`serverAddress`, `metricName`,
  `query`, `threshold`, `activationTargetQueryValue`). They are no
  longer referenced by the template.
- Keep `pollingInterval` and `cooldownPeriod`.
- Add the `nats.*` block.
- Bump `maxReplicas` from 8 to 20 to match what the external
  scaler had been configured for.

### Target GKE playbook shape

After ops #115 the GKE provision playbook should already have
`noetl_worker_autoscaling_enabled: false` and a comment block
explaining why. For this round, the GKE profile needs to flip
**keda** on while keeping the chart CPU HPA off:

- Add a new workload var:
  ```yaml
  # GKE workers scale via the Helm-NATS-fed KEDA ScaledObject
  # rendered by repos/ops/automation/helm/noetl/templates/worker-keda-scaledobject.yaml.
  # Both flags must agree: autoscaling.enabled=true gates the whole
  # autoscaling block; keda.enabled=true selects the NATS trigger
  # path over the chart CPU HPA. The chart guards make the two
  # templates mutually exclusive.
  noetl_worker_autoscaling_enabled: true
  noetl_worker_autoscaling_keda_enabled: true
  ```
  (Revert the `noetl_worker_autoscaling_enabled: false` from #115
  to `true`, since the keda branch now needs the umbrella enabled.)
- Update the helm-upgrade line to pass both flags:
  ```
  --set worker.autoscaling.enabled={{ workload.noetl_worker_autoscaling_enabled }}
  --set worker.autoscaling.keda.enabled={{ workload.noetl_worker_autoscaling_keda_enabled }}
  ```
